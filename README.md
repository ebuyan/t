# tportfolio

Выгрузка портфеля из T-Invest API в файлы Obsidian. Один бинарник без внешних
зависимостей, две независимые задачи по своим расписаниям:

| Задача | Что делает | Когда |
| --- | --- | --- |
| Реестр доходности | Добавляет запись в начало `Реестр.md` | Пн и Пт в 11:00 |
| Срез долей | Добавляет столбец в таблицы `Портфель.md` | 1 число квартала |

Задачи работают в отдельных горутинах: срез долей обходит два десятка инструментов
и не должен сдвигать запись в реестр.

## Реестр доходности (`Реестр.md`)

Дописывает в **начало** файла запись в формате inline-полей Dataview:

```
- date:: 2026-07-20
  income:: -1213000
  stock:: -1112000
  gold:: -101000
```

- `stock` — абсолютная доходность по акциям за всё время
- `gold` — то же по золоту
- `income` — их сумма

Источник — `positions[].expectedYield` из `GetPortfolio`, разбитый по классам
активов. Готового поля с абсолютной прибылью в ответе нет; сходимость проверена на
реальном портфеле: сумма, делённая на вложенное (кол-во × средняя цена), даёт ровно
тот процент, что API отдаёт в `expectedYield`.

Числа округляются **вверх до тысячи** — математически, то есть к нулю для
отрицательных: `-1 162 021 → -1 162 000`, но `1 162 021 → 1 163 000`. `income`
считается уже от округлённых слагаемых, иначе в файле нарушился бы инвариант
`income = stock + gold`.

Повторный запуск в тот же день обновляет запись на месте, а не добавляет вторую.

## Срез долей (`Портфель.md`)

Раз в квартал дописывает столбец во все таблицы файла с долями активов.
Таблицы опознаются по заголовку первого столбца:

| Таблица | Что считается |
| --- | --- |
| `Актив` | Доли акций и золота, плюс итог в рублях |
| `Компания` | Доля каждой бумаги от общей базы |
| `Сектор` | Доли секторов, нормированные к акциям (в сумме 100%) |
| `Дивиденды` | Столбец текущего года, обновляется на месте |

### База расчёта

Доли считаются от **акции + золото**. Рублёвый кэш в знаменатель не входит — так
доли «Акции» и «Золото» дают ровно 100%, как в файле исторически.

Золото — это `GLDRUB_TOM`: API отдаёт его как `instrumentType: "currency"`, то есть
внутри `totalAmountCurrencies`, отдельного класса активов в контракте нет. Список
таких тикеров — `goldTickers` в `snapshot.go`.

Дивидендная доходность — сумма полей `yieldValue` по всем выплатам `GetDividends`
за календарный год. Это доходность к цене закрытия на дату объявления, и ровно её
показывает приложение Т-Инвестиций — цифры в таблице совпадают с приложением.

Считать самим (сумма выплат к текущей цене) было бы не то же самое: после отсечки
цена падает, и доходность к ней получается выше. Пример на 20.07.2026, SBERP:
выплата 37,64 ₽ при цене объявления 279,92 ₽ даёт 13,44% (как в приложении), а к
текущей цене 258,67 ₽ вышло бы 14,55%.

### Сопоставление строк

Строки сопоставляются с позициями по тикеру — части подписи до `—`. Написанные
вручную названия при этом не перезаписываются названиями из API: в существующую
строку просто добавляется значение в новый столбец.

Отсюда требование: тикер в файле должен совпадать с тикером Т-Банка. Если написать
`SBER`, когда в портфеле лежит `SBERP`, строка не сопоставится и рядом появится
вторая — с тем же активом и пустой историей.

Повторный запуск в тот же день не плодит столбцы: существующий столбец с той же
датой обновляется на месте. Строки выбывших активов получают `—`, новые активы
дописываются в конец таблицы.

## Безопасность записи

Оба файла живут в синкающемся волте, поэтому запись устроена одинаково:

1. Файл читается и разбирается; всё, что не таблица и не запись, не трогается.
2. Рядом кладётся бэкап `<имя>.bak-<дата>` — отключается через `TINVEST_BACKUP=false`.
   Обсидиан такие файлы не показывает (расширение не `.md`), но они синкаются и
   копятся, так что после обкатки их стоит выключить.
3. Новое содержимое пишется во временный файл в том же каталоге и переносится
   поверх через `rename` — оборванная запись не оставит половину файла.

Тот же каталог для временного файла обязателен: `rename` атомарен только в пределах
одной файловой системы.

## Переменные окружения

| Переменная | Обяз. | Описание |
| --- | :---: | --- |
| `TINVEST_TOKEN` | ✓ | Токен T-Invest API (достаточно read-only) |
| `TINVEST_REGISTRY_FILE` |  | Реестр доходности (`Реестр.md`) |
| `TINVEST_REGISTRY_SCHEDULE` |  | Когда обновлять реестр: `Mon,Fri 11:00` или `11:00` для ежедневного |
| `TINVEST_PORTFOLIO_FILE` |  | Файл с таблицами долей (`Портфель.md`) |
| `TINVEST_PORTFOLIO_SCHEDULE` |  | Время `HH:MM` обновления долей первого числа квартала |
| `TINVEST_ACCOUNTS` |  | Id счетов через запятую (по умолчанию — все брокерские и ИИС) |
| `TINVEST_LIST_ACCOUNTS` |  | `true` — показать доступные счета и выйти |
| `TINVEST_NOW` |  | `true` — выполнить обе задачи сразу и выйти |
| `TINVEST_BACKUP` |  | `false` — не класть `.bak-` копии рядом с файлами (по умолчанию кладёт) |
| `TZ` |  | Часовой пояс для расписания, напр. `Europe/Moscow` |

Если не задано ни одно расписание — приложение делает один прогон тех задач, для
которых задан файл, и выходит. Это же поведение у `TINVEST_NOW`.

Ошибка отдельного прогона логируется, но не роняет процесс: иначе
`restart: unless-stopped` увёл бы контейнер в цикл перезапусков.

Токен выпускается в приложении Т-Инвестиций: *Настройки → Токен для T-Invest API*.
Достаточно токена **только для чтения** — приложение не делает торговых вызовов.

## Сборка и запуск

```sh
go build -o tportfolio .
# кросс-компиляция под сервер (Linux x86-64):
make build
```

```sh
export TINVEST_TOKEN=…

# посмотреть доступные счета и их id
TINVEST_LIST_ACCOUNTS=true ./tportfolio

# разовый прогон реестра
TINVEST_REGISTRY_FILE="/mnt/vault/Финансы/Портфель/Реестр.md" ./tportfolio

# конкретные счета
TINVEST_ACCOUNTS=2000000000,2000000001 ./tportfolio
```

Перед первым запуском на живом файле стоит прогнать на копии и посмотреть diff.

## Docker

```sh
docker build -t tportfolio .
docker run --rm -e TINVEST_TOKEN=… tportfolio
```

### В стеке ha (192.168.0.108)

Блок для `~/ha/docker-compose.yaml` — в стиле сервиса `energo`:

```yaml
  tportfolio:
    build: https://github.com/ebuyan/t.git#main
    container_name: tportfolio
    restart: unless-stopped
    environment:
      TINVEST_TOKEN: "…"
      TINVEST_REGISTRY_FILE: "/vault/Финансы/Портфель/Реестр.md"
      TINVEST_REGISTRY_SCHEDULE: "Mon,Fri 11:00"
      TINVEST_PORTFOLIO_FILE: "/vault/Финансы/Портфель/Портфель.md"
      TINVEST_PORTFOLIO_SCHEDULE: "10:00"
      TZ: "Europe/Moscow"
    volumes:
      - ./data:/vault
```

`./data` — тот же каталог, который раздаёт `dufs` на `file.buyan.fun`, то есть волт,
синкающийся из Obsidian по WebDAV. Пути в переменных указываются **внутренние**
(от `/vault`). Если волт лежит в подкаталоге `./data`, монтирование стоит сузить до
него — контейнеру не нужен доступ на запись ко всему файловому серверу.

`TZ` обязателен: в контейнере время идёт по UTC, и без него расписание сработает
на три часа раньше.

```sh
docker compose up -d --build tportfolio
docker compose logs -f tportfolio
```

Разовая проверка без ожидания расписания:

```sh
docker compose run --rm -e TINVEST_NOW=true tportfolio
```

## TLS и сертификат НУЦ Минцифры

`*.tbank.ru` выпущен удостоверяющим центром Минцифры («Russian Trusted Root CA»),
которого **нет** в стандартных хранилищах доверия — ни в alpine, ни в macOS. Без него
любой запрос падает с `x509: certificate signed by unknown authority`.

Корневой сертификат вшит в бинарник (`russian_trusted_root_ca.pem`, `go:embed`) и
добавляется **только в пул этого HTTP-клиента** — системное хранилище не изменяется,
на остальные приложения хоста доверие не распространяется (`client.go`).

- Источник: <https://gu-st.ru/content/Other/doc/russiantrustedca.pem>
- SHA-256: `D26D2D0231B7C39F92CC738512BA54103519E4405D68B5BD703E9788CA8ECF31`
- Срок действия корня: до 27.02.2032

Отпечаток стоит сверить при обновлении файла:

```sh
openssl x509 -in russian_trusted_root_ca.pem -noout -subject -fingerprint -sha256
```
